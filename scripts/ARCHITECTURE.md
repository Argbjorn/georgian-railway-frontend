# Scripts — архитектура

## make_routes_files.py

**Что производит:** `routes-list.js` (используется фронтом для отрисовки карты) и `routes.json` (используется для генерации статических страниц маршрутов).

**Источник данных:** Google Sheets через `GeoRoutesExcelHandler`. Таблица содержит два типа листов:
- лист `routes` — главный список всех маршрутов с метаданными;
- отдельные листы с номером маршрута в названии (например, `11`) — расписание конкретного маршрута со станциями и временем.

---

### Логика работы

**1. Фильтрация маршрутов**

Скрипт проходит по всем строкам листа `routes`. Маршруты с `show_on_site = 'n'` пропускаются полностью — они не попадают ни в какой выходной файл. Остальные обрабатываются вне зависимости от поля `active`.

**2. Метаданные маршрута**

Для каждого маршрута собираются:
- названия на трёх языках (ka/en/ru) — из них вырезаются паттерны вида «Поезд №11:», «Train #11:» и т.д.;
- флаги: `active`, `complete`, `online`, `online_tickets_current_site`, `online_tickets_new_site`;
- `frequency` — текстовое описание периодичности (например, «ежедневно»), пустая строка → `null`;
- `every_second_day_start` и `end_date` — даты начала/окончания сезонных или через-день маршрутов, хранятся как Unix timestamp в миллисекундах (UTC); могут быть `null`;
- `train_type` — тип поезда, пустая строка → `null`;
- `has_arrival_time` — флаг, влияет на то, в каком формате хранится расписание в листе маршрута.

**3. Станции и расписание**

Если у маршрута есть отдельный лист — из него берутся станции с временем. Неактивный маршрут без листа допустим — он попадёт в выходной файл без `stations`. Активный маршрут без листа — ошибка.

Для каждой станции:
- название на трёх языках берётся из `stations-list.js` по коду станции (код — это английское название, приведённое к нижнему регистру и без пробелов);
- роль: `start` (первая), `end` (последняя), `middle` (все остальные);
- время: лист может хранить либо одно поле `time` (простое расписание), либо два — `departure_time` и `arrival_time`. В первом случае `arrival_time` и `stop_time` будут `null`. Во втором — `stop_time` вычисляется как разница между прибытием и отправлением в минутах. Если время отсутствует — ставится `-`.

**4. Время в пути**

`travel_time` вычисляется только для маршрутов с `active = 'y'`. Берётся время отправления с первой станции и прибытия (или отправления, если `arrival_time` у второй станции `null`) на последней. Результат — строка формата `HH:MM`. Если данных нет — `-`.

**5. Цена**

Из четырёх возможных полей (`price_2_class`, `price_1_class`, `price_business`, `price_standard`) берутся только непустые. Если значений нет — `null`. Если одно — `{"price_type": "exact", "price": X}`. Если несколько — `{"price_type": "from", "price": MIN}` (минимальная из заполненных).

**6. Маршруты-аналоги**

Поле `analogue` — это ссылки на другие маршруты (номера через запятую), которые едут по тому же направлению. В выходной файл попадают только те аналоги, чей маршрут сам является `active = 'y'`. Неактивные аналоги молча отбрасываются.

**7. Сортировка и вывод**

Итоговый массив маршрутов сортируется по номеру (`ref`) и записывается в оба выходных файла. В `routes-list.js` добавляется JS-заголовок `export const routes = `.

## make_routes_geo_files.py

**Что производит:** набор JSON-файлов в `static/data/routes_geodata/`, по одному на каждый маршрут. Каждый файл содержит геоданные маршрута из OSM — координаты линий и остановок. Используются фронтом для непосредственной отрисовки маршрутов на карте.

**Источники данных:** Google Sheets (список маршрутов) + Overpass API (геоданные).

---

### Логика работы

**1. Какие маршруты обрабатываются**

Берутся только **активные** маршруты (`active = 'y'`) из листа `routes`.

**2. Запрос к Overpass**

Все OSM-идентификаторы активных маршрутов запрашиваются за один батч-запрос. Overpass возвращает данные в формате GeoJSON-подобного JSON с полной геометрией (`out geom`).

**3. Сохранение и очистка файлов**

Каждый маршрут сохраняется в отдельный файл. Имя файла — **OSM relation ID** маршрута (не номер маршрута `ref`), например `12345678.json`. Файл перезаписывается только если содержимое изменилось — это позволяет видеть в логе что реально обновилось за запуск.

После записи скрипт удаляет из папки все `.json`-файлы, чей OSM ID не входит в список активных маршрутов. Если Overpass вернул меньше маршрутов, чем ожидалось, скрипт выводит предупреждение — старый файл при этом остаётся.

## make_stations_files.py

**Что производит:** `stations-list.js` (используется фронтом для отрисовки станций на карте) и `stations.json` (используется `make_routes_files.py` для получения названий станций).

**Источники данных:** Overpass API (список станций и их координаты), `routes.json` (маршруты через каждую станцию), `stations.json` (существующие описания станций), `station_gr_codes.json` (коды ГЖД).

**Важная зависимость:** скрипт читает `routes.json`, поэтому `make_routes_files.py` должен быть запущен первым.

---

### Логика работы

**1. Какие станции попадают в список**

Список станций не берётся из таблицы напрямую. Вместо этого:
- из Overpass запрашиваются геоданные всех маршрутов с `show_on_site = 'y'`;
- из ответа извлекаются уникальные OSM node ID всех остановок (`role = 'stop'`);
- по этим ID делается второй запрос к Overpass — уже за данными самих станций.

Таким образом, источник правды о составе станций — **OSM**, а не таблица.

**2. Названия и код станции**

Для каждой станции берутся `name:en`, `name:ka`, `name:ru` из OSM-тегов. Если тег отсутствует — подставляются заглушки («unknown station» и т.д.).

Код станции (`code`) — это английское название, приведённое к нижнему регистру, без пробелов и дефисов. Именно этот код используется везде как идентификатор станции.

Если у двух OSM-нод одинаковый code (например, дублирующиеся узлы одной платформы), они объединяются в одну запись с массивом `id`.

**3. Тип станции**

Тип определяется жёстко по коду:
- `tbilisicentralstation` → `main`
- `batumicentral` → `beach`
- `kutaisiinternationalairport` → `airport`
- все остальные → `secondary`

**4. Код ГЖД**

Для каждой станции ищется числовой код в `station_gr_codes.json` (сопоставление по `name:en`). Для ряда станций, у которых название в OSM не совпадает с названием в справочнике ГЖД, есть жёстко прописанный словарь исключений (`exceptional_stations`). Если код не найден — ставится `00000`.

**5. Описания станций**

Описания не хранятся в Google Sheets и не приходят из OSM. Их первоисточник — само `stations.json`. Перед перезаписью файла скрипт читает из него существующие описания и добавляет их обратно в новые данные. Без этого шага описания бы терялись при каждом обновлении.

**6. Маршруты через станцию**

Из уже готового `routes.json` строится словарь: для каждой станции — какие маршруты через неё проходят и в какой роли (`departure` / `arrival` / `via`), с временем отправления/прибытия. Эти данные добавляются в запись станции как поле `routes` и используются на страницах станций.

**7. Сортировка и вывод**

Массив станций сортируется по `name_en` и записывается в оба файла. В `stations-list.js` добавляется JS-заголовок `export const stations = `.

Скрипт при запуске делает до 5 попыток с паузой 3 секунды между ними — Overpass API бывает нестабилен.

## check_all_routes.py

**Что делает:** инструмент ручной проверки — сравнивает список станций на каждом маршруте в таблице и в OSM, выводит расхождения в терминал. Ничего не записывает и не изменяет.

**Источники данных:** Google Sheets (локальные данные о станциях маршрута) + Overpass API (актуальные данные из OSM).

---

### Логика работы

Запускается в CI перед `make_routes_geo_files.py`. Расхождения выводятся в лог, но не останавливают билд.

Скрипт проходит по всем **активным** маршрутам с листом в таблице (`active = 'y'`) и за два батч-запроса к Overpass сопоставляет списки станций:

- **локальный** — коды станций из листа маршрута в таблице;
- **актуальный** — коды станций, полученные из OSM через Overpass: берётся relation маршрута, из его members извлекаются узлы с `role = 'stop'`, их `name:en` превращается в коды.

Если списки совпадают — маршрут помечается как OK. Если нет — выводятся две группы расхождений:
- станции, которые есть только в таблице (возможно, удалены из OSM или переименованы);
- станции, которые есть только в OSM (возможно, добавлены в OSM, но не занесены в таблицу).

**Порядок станций не проверяется** — только состав. Маршруты без листа в таблице автоматически пропускаются. Если все маршруты консистентны — выводится одно итоговое сообщение; при расхождениях — детали по каждому проблемному маршруту.
